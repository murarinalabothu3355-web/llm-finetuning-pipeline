# LLM Fine-Tuning & Deployment Pipeline

A LoRA/PEFT fine-tuning pipeline for instruction-tuning causal language models, with experiment tracking, containerized serving, and basic output validation — built around the Hugging Face `transformers` + `peft` stack.

## ⚠️ Important note on testing — please read

This repo was built in a sandboxed environment with **no internet access, no GPU, and none of the ML libraries (`torch`, `transformers`, `peft`) installed**. Because of that constraint, I was upfront with myself about what I could and couldn't verify, and split the codebase accordingly:

**✅ Actually executed and verified working in development:**
- `src/config.py` — configuration dataclasses
- `src/prepare_data.py` — dataset generation (run it, it produces real output)
- `src/train.py --dry-run` — config validation, data loading, train/eval split, prompt formatting
- `validate_output()` in `src/inference.py` — unit tested with 4 cases, all passing
- `src/mlflow_tracking.py --check` — structural check of the tracking wrapper

**📝 Written against stable, documented APIs but NOT executed (requires GPU/internet/libraries I didn't have access to):**
- The actual LoRA training loop in `train.py` (`run_training()`)
- `src/inference.py`'s model loading and generation
- `src/serve.py`'s FastAPI endpoints

**What this means for you:** the architecture, config, and data pipeline are solid and tested. Before you rely on the actual fine-tuning step, run:

```bash
pip install -r requirements.txt
python src/train.py --dry-run   # already verified — sanity-checks everything except the ML libraries
python src/train.py             # the real training run — please verify this works for you and open an issue/fix if any API has shifted
```

I'd rather tell you this clearly than hand you something that looks finished but might have a rough edge I couldn't catch.

## What this project does

1. **Prepares instruction-tuning data** (`src/prepare_data.py`) — ships with a small synthetic dataset (ML/infra Q&A pairs) so the pipeline runs standalone.
2. **Fine-tunes with LoRA** (`src/train.py`) — uses `peft.LoraConfig` to freeze the base model and train only small low-rank adapter matrices, drastically cutting trainable parameters and memory use.
3. **Tracks experiments with MLflow** (`src/mlflow_tracking.py`) — optional, logs hyperparameters and metrics for reproducibility.
4. **Validates outputs** (`src/inference.py`) — basic checks (empty/too-short generations, prompt echoing) before returning a result.
5. **Serves the model** (`src/serve.py` + `Dockerfile`) — a small FastAPI app, containerized for deployment.

## Why a small base model

The default config uses `distilgpt2` (~82M params) instead of a 7B+ model like LLaMA or Mistral, specifically so this pipeline can run end-to-end **on a laptop CPU** without needing a GPU rental. The LoRA/PEFT technique itself is identical regardless of model size — swap `base_model_name` in `src/config.py` to any causal LM on the Hugging Face Hub (e.g. `meta-llama/Llama-3.2-1B`) once you have GPU access, and everything else works unchanged.

## Quick start

```bash
git clone https://github.com/<your-username>/llm-finetuning-pipeline.git
cd llm-finetuning-pipeline
pip install -r requirements.txt

python src/prepare_data.py              # generates data/instructions.jsonl
python src/train.py --dry-run           # validates everything except the ML libraries
python src/train.py                     # full LoRA fine-tuning run
python src/inference.py --prompt "What does a UPS do in a data center?"
```

## Serving with Docker

```bash
docker build -t llm-finetuning-pipeline .
docker run -p 8000:8000 llm-finetuning-pipeline

curl -X POST http://localhost:8000/generate \
    -H "Content-Type: application/json" \
    -d '{"instruction": "What is LoRA?"}'
```

## Enabling MLflow tracking

Set `report_to: str = "mlflow"` in `TrainingConfig` (`src/config.py`), install `mlflow`, then run:

```bash
mlflow ui   # in a separate terminal, view at http://localhost:5000
python src/train.py
```
